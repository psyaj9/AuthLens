## Turn it into a more novel real-use platform

The best adaptation I’d build is:

# **PriorAuth Evidence Copilot**
### AI platform that prepares prior authorization and appeal packets for clinics

Instead of “ask questions about medical PDFs,” the platform solves a real workflow problem: clinics spend lots of time reading payer policy documents, checking whether the patient chart contains required evidence, finding missing documentation, and drafting prior authorization or appeal letters.

This is a strong use case because prior authorization is a documented burden: the AMA’s 2025 survey reports that **95% of physicians report care delays from prior authorization**, and physicians/staff spend an average of **13 hours per week** completing prior authorizations. ([American Medical Association](https://www.ama-assn.org/system/files/prior-authorization-survey.pdf)) CMS has also finalized rules to improve prior authorization data exchange, including payer APIs and requirements for prior authorization status, documentation requirements, decisions, and denial reasons. ([CMS](https://www.cms.gov/initiatives/burden-reduction/overview/interoperability/policies-regulations/cms-interoperability-prior-authorization-final-rule-cms-0057-f))

The key difference: your app would not be a “medical advice chatbot.” It would be an **evidence assembly and documentation workflow tool** for clinicians/admin staff, with human approval before anything is submitted.

## The product idea

A clinic user uploads or connects:

1. **Payer policy PDFs**  
   Example: insurer rules for MRI, biologic medication, CPAP, surgery, genetic test, physical therapy, etc.

2. **Patient packet**  
   Progress notes, lab reports, imaging reports, referral letters, denial letters, and medication history. For a demo, use synthetic/de-identified data.

3. **Requested service**  
   Example: “lumbar spine MRI,” “adalimumab prior auth,” “sleep study,” “cardiology referral,” or “appeal denied MRI request.”

The system outputs:

- **Approval criteria extracted from payer policy**
- **Checklist of required evidence**
- **Matched evidence from patient documents**
- **Missing documentation**
- **Risk of denial**
- **Draft prior authorization letter**
- **Draft appeal letter if denied**
- **Citations to exact policy and chart passages**
- **Human-review approval screen**

That is much more novel than a generic medical chatbot because it turns RAG into a real operational workflow.

## Safety and compliance boundaries

Do **not** position the first version as a diagnostic assistant. Position it as:

> “A documentation and evidence-preparation tool for clinician-reviewed prior authorization workflows.”

That matters because FDA guidance distinguishes some clinical decision support software from regulated device software depending on intended use, user, and whether the user can independently review the basis for the recommendation. ([U.S. Food and Drug Administration](https://www.fda.gov/regulatory-information/search-fda-guidance-documents/clinical-decision-support-software))

For any real patient data, you would need proper privacy controls. HIPAA’s Privacy Rule protects medical records and individually identifiable health information, sets limits on uses/disclosures, and gives individuals rights over their protected health information. ([HHS.gov](https://www.hhs.gov/hipaa/for-professionals/privacy/index.html)) For your portfolio/demo version, use synthetic or de-identified data only.

## Most important warning: the repo may need fixing before you build on it

There is one practical issue I noticed in the code.

The upload code chunks documents, embeds the chunk text, and sends vectors plus metadata to Pinecone. But the metadata appears to come from `chunk.metadata`, not from a metadata object that includes the actual text content. ([GitHub](https://raw.githubusercontent.com/snsupratim/medicalAssistant/main/server/modules/load_vectorstore.py)) Then the question-answering route later tries to rebuild documents using `match["metadata"].get("text", "")`. ([GitHub](https://raw.githubusercontent.com/snsupratim/medicalAssistant/main/server/routes/ask_question.py))

That means the retrieved document content may come back empty unless the text was added into metadata elsewhere. Before adapting it, you should fix the upsert step so each Pinecone vector stores something like:

```python
metadata = {
    "text": chunk.page_content,
    "source": file_path,
    "page": chunk.metadata.get("page"),
    "document_type": "payer_policy",
    "case_id": case_id
}
```

Without this, your RAG answers may be weak or broken because the LLM will not receive the actual retrieved text.
